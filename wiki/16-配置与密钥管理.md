# 配置与密钥管理

本文档说明 Trae Social Android 项目的配置与密钥管理机制。项目将配置分为敏感（API Key 等）与非敏感两类，分别用 `EncryptedSharedPreferences` 与 `DataStore Preferences` 存储。

## 两级存储策略

| 类别 | 存储方式 | 文件名 | 内容 |
| --- | --- | --- | --- |
| 敏感配置 | EncryptedSharedPreferences | `social_secure_prefs` | API Key / Base URL / 模型名（按 provider id） |
| 非敏感配置 | DataStore Preferences | `social_prefs` | 默认 provider / 引导完成 / 引导跳过 / AI 活跃度档位 |

## EncryptedSharedPreferences

- 限定符 `@SecurePreferences`（`@Qualifier BINARY`）。`@Singleton`，由 `ConfigRepository` 通过 `@SecurePreferences` 注入。
- `MasterKey` 基于 Android Keystore `AES256_GCM`。加密方案 `AES256_SIV`（key）+ `AES256_GCM`（value）。
- key 命名规则：`api_key_${provider.id}` / `base_url_${provider.id}` / `model_${provider.id}`。
- 三级降级容错：
  1. 创建失败。
  2. `deleteSharedPreferences` 删除损坏文件重建。
  3. 仍失败回退普通 `SharedPreferences`（文件名 `social_secure_prefs_fallback`，`MODE_PRIVATE`）。
  - 每级 `runCatching.getOrElse` + `Timber.e`。
- `apiKeyPreview(provider)` 脱敏：长度 `<=8` 返回 `"***"`，否则 `"首4***末4"`，`withContext(Dispatchers.IO)`。

## DataStore Preferences

- `Context.socialDataStore`（`preferencesDataStore name="social_prefs"`）。
- Preferences Keys（`ConfigRepository` companion）：
  - `KEY_DEFAULT_PROVIDER = stringPreferencesKey("default_provider")`
  - `KEY_ONBOARDING_COMPLETED = booleanPreferencesKey("onboarding_completed")`
  - `KEY_ONBOARDING_SKIPPED = booleanPreferencesKey("onboarding_skipped")`
  - `KEY_AI_ACTIVITY_LEVEL = stringPreferencesKey("ai_activity_level")`

## ConfigRepository 方法清单

| 方法 | 存储 | 说明 |
| --- | --- | --- |
| getApiKey/setApiKey | Encrypted | 按 provider |
| getBaseUrl/setBaseUrl | Encrypted | 按 provider |
| getModelName/setModelName | Encrypted | 按 provider |
| apiKeyPreview | Encrypted | 脱敏预览 |
| getDefaultProvider/setDefaultProvider | DataStore | key `default_provider` |
| isOnboardingCompleted/setOnboardingCompleted | DataStore | key `onboarding_completed` |
| isOnboardingSkipped/setOnboardingSkipped | DataStore | key `onboarding_skipped`（区分跳过与完成） |
| getAiActivityLevel（默认 MEDIUM）/setAiActivityLevel | DataStore | key `ai_activity_level` |
| activityLevelChanges | SharedFlow | `setAiActivityLevel` 后 `tryEmit`，`SchedulerInitializer` 收集 |

## LlmConfigProvider 桥接

- `core-llm` 定义 `LlmConfigProvider` 接口（全 `suspend`），`app` 模块 `AppLlmConfigProvider` 实现并桥接到 `ConfigRepository`。
- 全 `suspend` 避免 `runBlocking` ANR（IMPL-14）。`AuthInterceptor` 运行在 OkHttp 调度线程，可安全 `runBlocking` 调用。
- `getDefaultProvider` 默认回退 `OPENAI`。

## LlmCacheInvalidator

- `core-data` 定义 `fun interface LlmCacheInvalidator { suspend fun invalidateCache() }`（避免 feature 模块依赖 `core-llm`）。
- `app` 模块 `AssetProviderModule` 提供 lambda 实现 -> `LlmProviderRegistry.invalidateCache()`。
- 触发时机：
  - `ApiKeyManagementScreen` 保存 Key/URL/模型后。
  - `OnboardingViewModel.testConnection`/`saveAndComplete`。
  - 切换默认 provider。

## 配置变更影响链

### API Key 变更链

1. 用户在 `ApiKeyManagementScreen` 保存 API Key。
2. `ApiKeyViewModel.setApiKey`。
3. `ConfigRepository.setApiKey`（写 Encrypted）。
4. `cacheInvalidator.invalidateCache()`。
5. `LlmProviderRegistry.invalidateCache()`（清 `clients` ConcurrentHashMap）。
6. 下次 `getClient` 重新 `createClient`。

### AI 活跃度变更链

1. 用户切换 AI 活跃度档位。
2. `ProfileViewModel.setAiActivityLevel`。
3. `ConfigRepository.setAiActivityLevel`（写 DataStore）。
4. `activityLevelChanges.tryEmit`。
5. `SchedulerInitializer.observeActivityLevelChanges` 收集。
6. REPLACE 重排 `PersonaUpdateWorker` + `rateLimiter.reconfigure(level)`。

## 引导状态

- `isOnboardingCompleted`：boolean，完成引导（含跳过）后 `true`。`MainActivity` 用 `produceState` 异步读取决定起始路由（`MAIN` 或 `ONBOARDING`）。
- `isOnboardingSkipped`：boolean，区分"跳过"与"完成"。跳过时 `FeedScreen` 顶部 `OnboardingSkippedBanner` 提示补全配置。
- 引导跳过后仍可在 我的 -> 设置 -> API Key 管理 补全。

## UserConfigEntity（KV 表）

- `user_configs` 表 PK `key`，KV 结构。
- 注意：敏感数据不走此表，走 `EncryptedSharedPreferences`（RISK-11）。
- 当前主要用于一些非敏感的持久化 KV。
