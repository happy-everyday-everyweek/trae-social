# 配置与密钥管理

本文档说明 Trae Social Android 项目的配置与密钥管理机制。项目将配置分为敏感（API Key 等）与非敏感两类，分别用 `EncryptedSharedPreferences` 与 `DataStore Preferences` 存储。

> **#151 重构**：旧 EncryptedSharedPreferences 按 provider 槽位寻址的配置抽象（`api_key_${provider.id}`）
> 改为按端点寻址（`api_key_ep_<endpointId>`）。端点元数据（baseUrl / model / capabilities / orderIndex）
> 迁移到 Room `llm_endpoints` 表（详见 [13 数据库设计](./13-数据库设计.md)）。
> 旧 key 保留不删（迁移幂等，回滚安全）。

## 两级存储策略

| 类别 | 存储方式 | 文件名 | 内容 |
| --- | --- | --- | --- |
| 敏感配置 | EncryptedSharedPreferences | `social_secure_prefs` | API Key（按 endpointId + 历史遗留按 provider id 旧槽位） |
| 非敏感配置 | DataStore Preferences | `social_prefs` | 引导完成 / 引导跳过 / AI 活跃度档位 / 用户行为建模配置 |
| 端点元数据 | Room（social.db `llm_endpoints` 表） | `social.db` | displayName / protocol / baseUrl / model / capabilities / orderIndex |

## EncryptedSharedPreferences

- 限定符 `@SecurePreferences`（`@Qualifier BINARY`）。`@Singleton`，由 `ConfigRepository` 通过 `@SecurePreferences` 注入。
- `MasterKey` 基于 Android Keystore `AES256_GCM`。加密方案 `AES256_SIV`（key）+ `AES256_GCM`（value）。
- key 命名规则：
  - **新命名空间（#151）**：`api_key_ep_<endpointId>`（端点 UUID，由 Room `llm_endpoints.id` 主键对应）。
  - **旧命名空间（保留不删，迁移用）**：`api_key_${provider.id}` / `base_url_${provider.id}` / `model_${provider.id}`。
- 三级降级容错：
  1. 创建失败。
  2. `deleteSharedPreferences` 删除损坏文件重建。
  3. 仍失败回退普通 `SharedPreferences`（文件名 `social_secure_prefs_fallback`，`MODE_PRIVATE`）。
  - 每级 `runCatching.getOrElse` + `Timber.e`。
- `endpointApiKeyPreview(endpointId)` 脱敏：长度 `<=8` 返回 `"***"`，否则 `"首4***末4"`，`withContext(Dispatchers.IO)`。

## DataStore Preferences

- `Context.socialDataStore`（`preferencesDataStore name="social_prefs"`）。
- Preferences Keys（`ConfigRepository` companion）：
  - `KEY_DEFAULT_PROVIDER = stringPreferencesKey("default_provider")`（迁移用，#151 后由 `orderIndex=0` 主端点取代）
  - `KEY_ONBOARDING_COMPLETED = booleanPreferencesKey("onboarding_completed")`
  - `KEY_ONBOARDING_SKIPPED = booleanPreferencesKey("onboarding_skipped")`
  - `KEY_AI_ACTIVITY_LEVEL = stringPreferencesKey("ai_activity_level")`
  - `KEY_ENDPOINT_MIGRATION_DONE = booleanPreferencesKey("endpoint_migration_done")`（#151 端点迁移幂等标记）
  - 用户行为建模相关键（#146）：`profiling_enabled` / `feedback_gray_ratio` / `event_ttl_days` / `profile_period_hours` / `feedback_agent_enabled` / `scenario_enabled_<id>`

## ConfigRepository 方法清单（#151 后）

### 端点 CRUD（新）

| 方法 | 存储 | 说明 |
| --- | --- | --- |
| `listEndpoints()` | Room + Encrypted | 触发旧 provider 迁移，返回按 `orderIndex` 升序的端点列表 |
| `observeEndpoints()` | Room | 暴露 `Flow<List<LlmEndpointEntity>>`，UI 用 |
| `getEndpoint(id)` | Room | 按 id 取端点 |
| `addEndpoint(displayName, protocol, baseUrl, model, capabilities, apiKey?)` | Room + Encrypted | 新建端点，`orderIndex` 取 max+1 |
| `updateEndpoint(id, displayName, protocol, baseUrl, model, capabilities)` | Room | 更新端点元数据（不含 API Key） |
| `deleteEndpoint(id)` | Room + Encrypted | 删除端点 + 清理其 API Key |
| `reorderEndpoints(orderedIds)` | Room | 事务重写 `orderIndex`（委托 `LlmEndpointDao.reorder` 的 `@Transaction` 默认方法，内含 shift + N 次 setOrderIndex） |
| `getEndpointApiKey(endpointId)` | Encrypted | 读取端点 API Key |
| `setEndpointApiKey(endpointId, key)` | Encrypted | 写入端点 API Key |
| `endpointApiKeyPreview(endpointId)` | Encrypted | 脱敏预览 |

## EndpointConfigProvider 桥接（#151）

- `core-llm` 定义 `EndpointConfigProvider` 接口（全 `suspend`），`app` 模块 `AppEndpointConfigProvider` 实现并桥接到 `ConfigRepository`。
- 全 `suspend` 避免 `runBlocking` ANR。
- `listEndpoints` 首次调用触发旧 provider 配置迁移（幂等）。
- `observeEndpointChanges()` 返回 `Flow<Unit>`，任何端点 CRUD / API Key 变更后发出 Unit，
  `EndpointRegistry` 收集后失效缓存的 SDK client 实例。

## LlmCacheInvalidator

- `core-data` 定义 `fun interface LlmCacheInvalidator { suspend fun invalidateCache() }`（避免 feature 模块依赖 `core-llm`）。
- `app` 模块 `AssetProviderModule` 提供 lambda 实现 -> `EndpointRegistry.invalidateCache()`。
- 触发时机：
  - `ApiKeyManagementScreen` 增删改端点 / 保存 API Key 后。
  - `OnboardingViewModel.testConnection` / `saveAndComplete`。
  - 端点 reorder。

## 配置变更影响链

### API Key 变更链（#151 后）

1. 用户在 `ApiKeyManagementScreen` 保存 API Key。
2. `ApiKeyViewModel.setApiKey`。
3. `ConfigRepository.setEndpointApiKey`（写 Encrypted `api_key_ep_<endpointId>`）。
4. `_endpointChanges.tryEmit(Unit)`。
5. `EndpointRegistry` 订阅 `observeEndpointChanges()` 收集后 `invalidateCache()`（清 `clients` ConcurrentHashMap）。
6. 下次 `getClient(endpointId)` 重新创建 SDK client。

### AI 活跃度变更链

1. 用户切换 AI 活跃度档位。
2. `ProfileViewModel.setAiActivityLevel`。
3. `ConfigRepository.setAiActivityLevel`（写 DataStore）。
4. `activityLevelChanges.tryEmit`。
5. `SchedulerInitializer.observeActivityLevelChanges` 收集。
6. REPLACE 重排 `PersonaUpdateWorker` + `rateLimiter.reconfigure(level)`。

## 引导状态

- `isOnboardingCompleted`：boolean，完成引导（含跳过）后 `true`。`MainActivity` 用 `produceState` 异步读取决定起始路由（`MAIN` 或 `ONBOARDING`）。
- `isOnboardingSkipped`：boolean，区分"跳过"与"完成"。跳过时 `FeedScreen` 顶部 `OnboardingSkippedBanner` 提示补全配置。
- 引导跳过后仍可在 我的 -> 设置 -> API Key 管理 补全。

## UserConfigEntity（KV 表）

- `user_configs` 表 PK `key`，KV 结构。
- 注意：敏感数据不走此表，走 `EncryptedSharedPreferences`（RISK-11）。
- 当前主要用于一些非敏感的持久化 KV。
